# DECISIONS: Checkout Split (parent)

Binding decisions for the recursive decomposition. This file was created in a retrofit on
2026-07-22: the original decomposition run predates the delivery-topology discipline, so the
entries below record the decision as adjudicated then, dated when it was written down, not
reconstructed interrogation history.

- 2026-07-22 (retrofit): Delivery topology, per subsystem. Both children (orders, payments) are
  declared ISOLATED: each team builds and tests without access to the other's service or a full
  multi-service platform environment. Consequence, per the machinery stand-in doctrine: each
  child's BUILD.md carries a "Neighbor stand-ins and test environment" section with one contract
  stand-in per neighboring boundary, specified by the pack's boundary event rows plus the
  neighbor's contract machine, which the parent supplies alongside the pack (this design's
  `contracts/OrdersContract.machine.json` and `contracts/PaymentsContract.machine.json` are those
  public artifacts). The stand-ins are held to conformance suites generated by `machinery oracle`
  over the contract machines, keyed on stable ids. Each child conductor re-asks this question at
  its own Phase 0 and may flip it if the team's access situation differs.
- 2026-07-22 (retrofit): Parent residuals stay parental. What the stand-ins and the refinement
  proofs cannot prove (end-to-end checkout latency, liveness across the two mutually dependent
  contracts, any channel outside the three modeled boundary events) is owned by the parent's
  cross-context assembly suite, named in the parent BUILD.md, and is not delegated to either
  child.

# Formal verification of the go-crm design

Every model here is checked exhaustively by TLC. Nothing is asserted by inspection.
Run the whole suite from the example root:

```sh
machinery verify-formal design    # regenerates everything from source, runs this design's proofs
```

(CI runs the same command across all six example designs.) Requires Java 11+; the runner
fetches a version-pinned, checksum-verified `tla2tools.jar` into `~/.cache/machinery` on
first use.

What "cannot drift" means here, precisely: the control-flow models are regenerated from
the machine JSON on every run; the data and refinement models are generated from
`Deal.semantics.yaml` only AFTER `machinery refine` reconciles that annotation against the
machine (states, transition structure, rollback routing), so an edit to either side
fails generation instead of proving the stale twin; and `System.tla` INSTANCEs the
generated `DealContract`, so the assumption it makes is the module the refinement proves.

What is NOT modeled at any rung here, stated once: guard semantics (erased, sound for
safety; liveness is conditional on guard-list exhaustiveness, which `machinery lint`
discharges via fallback-or-`_exhaustive`), real data and arithmetic beyond the refined
variables, multiple concurrent instances (two deals racing), and real time. Those live in
the named-unit contracts, the property tests, and the implementation suite. Each
generated module states its assumptions in its header.

## 1. Control-flow models (generated by `machinery tla`)

`Deal.tla`, `Task.tla`, `User.tla`, `Session.tla`, `CommandExecution.tla` are generated from
the machine JSON. The state graph is exact; guards are over-approximated. Every retry loop
gets its own bounded counter (CommandExecution's DBLocked resumes to either phase; the
model includes both resume targets). Each proves:

- `Live_OverlayResolves` (the substantive property): any transient overlay state always
  returns to a resting domain state. No stuck half-persist, no infinite retry.
- deadlock-freedom (final states stutter via `Terminated`).
- `TypeOK`: a regression lock (generator output satisfies it by construction; it guards
  against hand-edited specs).

## 2. Data-refined domain invariants (generated by `machinery refine` from `Deal.semantics.yaml`)

The control-flow model abstracts away the committed data. `machinery refine` reconciles
`Deal.semantics.yaml` against the machine (a drifted annotation is a hard generation
error), then generates `DealData.tla`, which proves the domain contract over the refined
state:

- `Inv_Atomic`: persist atomicity. The committed stage flips only on the atomic commit; any failure
  rolls back to the prior stage unchanged. No torn state is reachable.
- `StageForward`: the `deal-stage-forward` invariant (forward, to Won/Lost, or reopen only).
- `Inv_CloseDate`: the `deal-won-has-closedate` invariant. Plus well-formedness and liveness.

Still assumed (carried by the named-unit contracts and the implementation tests): the
pending/prior context updates the actions perform, and the retry bound.

## 3. Refinement (generated: `DealContract` + `DealRefinement`)

`DealContract.tla` is the abstract contract the big picture assumes of a deal aggregate: resting or
busy, atomic while busy, always terminating. `DealRefinement.tla` defines the mapping from the
concrete `DealData` state to the contract and TLC proves every concrete behavior is a contract
behavior (`RefSpec`, `RefTermination`). A subsystem is a correct part of the whole exactly when it
refines the contract its neighbors assumed.

## 4. Assume-guarantee composition (`System`)

`System.tla` composes a caller (the command envelope) with the deal aggregate known ONLY through
`DealContract`, INSTANCED directly: the aggregate's steps in the composition are the contract
module's own actions, so the assumption and the proven guarantee cannot drift apart silently.
TLC proves the caller's guarantee: a running command always completes (`Live_CommandCompletes`)
and leaves the deal resting (`Safe_DoneImpliesResting`), and additionally that the composition's
aggregate behavior satisfies the contract spec (`AggregateContractHolds`), which is what justifies
substituting the refinement-checked `DealData`. The parts are verified against small contracts and
never against the flattened whole. That is the rule that lets the approach scale to large systems.

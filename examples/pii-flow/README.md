# pii-flow: a Datalog external checker

This example is the reference implementation of machinery's
[external-checker contract](../../docs/external-checkers.md), built with a
real bring-your-own engine: [Soufflé](https://souffle-lang.github.io/), a
Datalog solver. It enforces one invariant machinery's built-in gates do not
express: a sensitive-data-flow property over the domain model's relationship
graph.

## What it demonstrates

- **A closed-world Datalog checker.** `rules.dl` derives `tainted(E)` from
  every entity that holds a sensitive attribute and everything it flows to,
  except through a `redacted` node. Soufflé's negation-as-failure
  (`!redacted(F)`) is what makes redaction a taint boundary: nothing derived
  downstream of a redactor is ever marked tainted. A `leak(E)` fact is derived
  only when a tainted entity is also a declared sink. Stratified negation
  keeps this decidable and deterministic.
- **The coverage-claim-plus-residuals mechanism.** The manifest claims all
  three `priv-*` invariants (`coverage.claim: ["priv-*"]`) but the checker
  only decides one of them from the flow graph. The other two,
  `priv-consent-required` and `priv-minimal-collection`, are declared
  **residuals** with a reason: they are process and design-review controls,
  not something a static graph analysis can decide. Coverage is a hard rule
  in machinery: a claimed invariant that is neither in the evidence nor a
  declared residual fails the gate. This example shows both halves of that
  rule working together on the same manifest.
- **A domain designed to pass.** The model is a straight-line pipeline,
  `DataSubject -> ProcessingActivity -> Redactor -> AnalyticsExport`, where
  the export sink is reachable only by passing through the redactor. No
  sensitive attribute can reach the sink unredacted, so the checker's verdict
  is `pass` and its evidence carries zero findings.

## File layout

```
examples/pii-flow/
  design/
    pii-flow.modelith.yaml               # the domain model (Modelith)
    checkers/
      pii-flow.checker.yaml              # the tool-neutral manifest (committed)
      pii-flow/
        rules.dl                         # the Datalog program (committed)
        adapter.py                       # projection -> facts -> Soufflé -> evidence
        projection.json                  # generated by `machinery project` (committed)
        evidence.json                    # produced by adapter.py via Soufflé (committed)
  checkers.local.example.yaml            # sample local registry (copy, do not commit as-is)
  README.md
```

`pii-flow.checker.yaml` names no binary and no engine; it declares only the
projection slice it needs (`model`, `invariants`, `relationships`), what it
claims, its residuals, and where its evidence lives. The `config` block
(`enforces_invariant`, `sensitive`, `sinks`, `redacted`) is opaque to
machinery and passed through verbatim to `adapter.py`, which is how the
checker supplies its own domain knowledge without machinery ever needing to
understand it.

## Why `machinery check --gate gk` is hermetic

The pure gate never runs Soufflé, never runs Python, and never shells out to
anything. It:

1. regenerates the projection from `pii-flow.modelith.yaml` and byte-matches
   it against the committed `projection.json`;
2. reads the committed `evidence.json` and checks that its `input_hash`
   equals the hash of that fresh projection (the binding: the verdict was
   computed over exactly this design, not a stale one);
3. checks `verdict == pass` and that every claimed `priv-*` invariant is
   either in `coverage` or a declared residual.

Because the projection and the evidence are both committed artifacts, this
check runs anywhere machinery runs, with no Soufflé or Python installation
required. That is the split this example exists to show: the deterministic
half lives in `machinery check`; the engine half is a separate, explicit
step.

## Enabling the engine phase

The gate above only verifies that committed evidence is fresh and covers
what it claims; it does not re-run Soufflé. To actually regenerate the
evidence, or to have machinery re-run the checker and confirm the verdict was
earned:

1. Copy the sample registry into the repo root (it is git-ignored, so each
   machine points at its own local engine):

   ```
   mkdir -p .machinery
   cp examples/pii-flow/checkers.local.example.yaml .machinery/checkers.local.yaml
   ```

2. Run the engine phase:

   ```
   machinery verify-checkers examples/pii-flow/design
   ```

   This regenerates the projection, runs `adapter.py` (which shells out to
   `souffle`), and confirms the freshly produced verdict matches the
   committed evidence.

To regenerate the committed artifacts by hand instead:

```
machinery project examples/pii-flow/design
python3 examples/pii-flow/design/checkers/pii-flow/adapter.py \
  examples/pii-flow/design/checkers/pii-flow/projection.json \
  <config.json> \
  examples/pii-flow/design/checkers/pii-flow/evidence.json
```

where `<config.json>` is the manifest's `config` block written out as its own
JSON file (machinery generates this file automatically during
`verify-checkers`; it is never committed).

## Prerequisites

- [Soufflé](https://souffle-lang.github.io/install/) (`souffle --version`)
- Python 3, standard library only (no `pip install` needed)

Neither is required to run `machinery check --gate gk`; both are required
only for `machinery verify-checkers` or for regenerating the evidence by
hand.
